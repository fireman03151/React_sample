## Testes de Aceitação do Usuário

Na engenharia e suas várias subdisciplinas, o teste de aceitação é um teste conduzido para determinar se os requisitos de uma especificação ou contrato são atendidos. Pode envolver testes químicos, testes físicos ou testes de desempenho.

Na engenharia de sistemas, pode envolver testes de caixa preta realizados em um sistema (por exemplo: um software, muitas peças mecânicas manufaturadas ou lotes de produtos químicos) antes de sua entrega.

No teste de software, o ISTQB define aceitação como: teste formal com relação às necessidades do usuário, requisitos e processos de negócios realizados para determinar se um sistema satisfaz os critérios de aceitação e para permitir que o usuário, clientes ou outra entidade autorizada determine se aceita ou não o sistema. O teste de aceitação também é conhecido como teste de aceitação do usuário (UAT), teste do usuário final, teste de aceitação operacional (OAT) ou teste de campo (aceitação).

Um teste de fumaça pode ser usado como um teste de aceitação antes de introduzir uma compilação de software no processo de teste principal.

Uma das etapas finais do teste de software é o teste de aceitação do usuário (UAT). O UAT garante que usuários reais possam usar o software. Também chamado de teste beta, aplicativo e usuário final.

O UAT verifica se tudo funciona corretamente e se não há falhas. Aqueles do público-alvo devem completar o teste; Isso pode ser composto de muitas pessoas envolvidas no processo e qualquer um que é capaz de testar antes de ir ao vivo. O feedback deste teste é encaminhado para a equipe de desenvolvimento para quaisquer alterações específicas.

#### Por que precisamos de UAT

*   Alterações de requisitos podem não ter sido comunicadas aos desenvolvedores
    
*   O software pode não estar entregando realmente o que significou para
    
*   Alguma lógica ou processo de negócios pode precisar da atenção do usuário
    

#### O que é necessário antes de começarmos o UAT

*   O requriement completo é assinado e está disponível como documentado
    
*   O código está em funcionamento ou em condição demo
    
*   UAT é ambiente está pronto para acesso
    
*   Não deve haver qualquer defeito que irá quebrar o código
    
*   Dados de teste preparados de acordo com o scneario AO VIVO
