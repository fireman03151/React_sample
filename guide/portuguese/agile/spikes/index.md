## Espigões

Nem tudo em que sua equipe trabalha durante o Sprint são Estórias de Usuário. Às vezes, uma história de usuário não pode ser efetivamente estimada ou a pesquisa é necessária. Talvez seja necessária ajuda para decidir entre diferentes arquiteturas, para descartar alguns riscos, ou pode ser uma Prova de Conceito (POC) para demonstrar uma capacidade.

Nestes casos, um Spike é adicionado ao Sprint Backlog. Os Critérios de Aceitação para o Spike devem ser uma resposta para a questão que está sendo colocada. Se o experimento é mais difícil do que o previsto, talvez a resposta seja negativa. (É por isso que o Spike deve ser limitado no tempo.)

O tempo e a energia investidos no Spike são limitados intencionalmente para que o trabalho possa ser concluído dentro do Sprint, enquanto outras Estórias de Usuário são minimamente impactadas. O Spike geralmente não recebe uma estimativa de ponto de história, mas recebe um número fixo de horas para ser trabalhado.

Demonstrar os resultados do Spike na Sprint Review. Com base nas novas informações, a pergunta original é reformulada como uma nova história de usuário em um futuro Sprint.

Seu time ganhou as informações necessárias para tomar melhores decisões; para reduzir a incerteza. Com uma quantidade pequena e medida de seus recursos para aumentar a qualidade e o valor do trabalho em Sprints futuros, em vez de tomar decisões baseadas em suposições.
